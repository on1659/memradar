# Search MVP — 상세 기획

> Memradar 를 "보는 툴"에서 "다시 찾는 아카이브"로 바꾸는 기능

현재 상태: 기본 풀텍스트 검색은 동작 (Phase A), 필터·정렬 일부는 진행 중이다. 상세 로드맵은 [ROADMAP.md §2.2](./ROADMAP.md) 참고.

## 컨셉

사용자가 쌓아둔 AI 대화 기록 안에서 원하는 작업·질문·파일·도구 사용 흔적을 빠르게 다시 찾을 수 있게 한다.

Code Report 가 처음의 "와"를 만드는 기능이라면, Search 는 계속 돌아오게 만드는 기능이다.

---

## 목표

- 특정 작업을 했던 세션을 빠르게 다시 찾기
- 예전에 어떤 프롬프트/질문/도구 조합을 썼는지 회고하기
- "이거 전에 어디서 했지?"를 해결하는 기본 탐색 도구 만들기

---

## MVP 범위

### 1. 풀텍스트 검색

- 세션 전체와 메시지 텍스트를 대상으로 검색
- 사용자 메시지와 assistant 메시지 모두 검색 대상
- 파일명, `cwd`, 모델명, tool use 이름도 검색 대상에 포함 가능

예시:

- `zustand`
- `useEffect`
- `vercel deploy`
- `read_file`
- `~/projects/my-app`

### 2. 필터

검색 결과를 아래 기준으로 좁힐 수 있어야 한다.

- 모델
- 툴 사용명
- 프로젝트 경로(`cwd`)
- 날짜 범위
- 역할
  - user만
  - assistant만
  - 전체

### 3. 결과 리스트

각 결과는 아래 정보를 포함한다.

- 세션 제목
- 매칭된 문장 또는 메시지 스니펫
- 일치 개수(hit count)
- 날짜
- 모델
- `cwd` 또는 프로젝트명

### 4. 세션 이동

- 결과 클릭 시 세션 상세 화면으로 이동
- 가능하면 해당 메시지 위치로 스크롤
- 검색어 하이라이트 표시

### 5. 정렬

초기 MVP는 아래 2가지만 있으면 충분하다.

- 관련도순
- 최신순

---

## UX 흐름

1. 상단 또는 대시보드에 검색 입력창 노출
2. 사용자가 키워드 입력
3. 결과 리스트 즉시 갱신
4. 필요 시 필터 추가 적용
5. 결과 클릭
6. 세션 상세에서 해당 맥락 확인

---

## UI 구성

### 검색 바

- placeholder 예시:
  - `메시지, 도구, 파일, 프로젝트 검색`
- debounce 적용
- 검색어가 비어 있으면 최근 세션 또는 인기 키워드 대신 기본 세션 리스트 유지 가능

### 필터 바

- 모델 드롭다운
- 툴 드롭다운 또는 다중 선택
- 프로젝트 드롭다운
- 날짜 범위 선택
- 역할 토글

### 결과 카드

- 첫 줄: 세션 제목
- 둘째 줄: 검색어가 포함된 스니펫
- 메타 정보:
  - 날짜
  - 모델
  - 프로젝트
  - hit count

---

## 검색 동작 규칙

### 매칭 규칙

- 대소문자 구분 없음
- 부분 일치 허용
- 여러 단어 입력 시 AND 검색을 기본으로 고려
  - 예: `react zustand` → 둘 다 포함된 결과 우선

### 스니펫 규칙

- 검색어 주변 1~2문장만 추출
- 너무 긴 메시지는 앞뒤를 잘라서 보여주기
- 검색어는 강조 스타일 적용

### 결과 그룹핑

초기 MVP는 세션 단위 또는 메시지 단위 중 하나만 선택해도 된다.

추천:

- 기본 표시는 메시지 단위
- 같은 세션 결과는 시각적으로 묶기

이유:

- 사용자는 정확히 어디에서 언급됐는지 바로 보고 싶어함
- 세션 단위만 보여주면 다시 찾는 속도가 떨어짐

## 기술 구현 방향

### 인덱싱

세션 파싱 후 검색용 인덱스를 만든다.

예시 구조:

```typescript
interface SearchRecord {
  sessionId: string
  messageIndex: number
  text: string
  role: 'user' | 'assistant'
  model?: string
  cwd?: string
  timestamp: string
  tools: string[]
}
```

초기 MVP는 별도 검색 엔진 없이도 구현 가능하다.

- 메모리 내 배열 필터링
- 세션 수가 늘면 IndexedDB 캐시와 함께 간단한 역색인 고려

### 상태 관리

- 검색어
- 필터 상태
- 정렬 기준
- 검색 결과

이 값들은 라우트 쿼리스트링과 연결되면 좋다.

예:

- `/search?q=zustand`
- `/search?q=deploy&model=claude-sonnet&tool=bash`

### 하이라이트

- 검색어 기준으로 문자열 분리 후 `<mark>` 렌더링
- escape 처리 필수

---

## 성능 기준

- 입력 후 결과 체감 반응은 즉시 또는 200ms 이내 목표
- 검색은 debounce 150~250ms 정도 권장
- 대용량 세션에서도 UI 멈춤이 느껴지지 않아야 함
- 필요 시 검색 전용 Web Worker는 후순위로 추가

---

## 제외 범위

초기 MVP에서는 아래 기능을 넣지 않는다.

- 벡터 검색
- 자연어 의미 검색
- AI 기반 검색어 추천
- 복잡한 쿼리 문법
- 정규식 검색
- 저장된 검색
- 검색 결과 공유 링크의 완전한 재현성 보장

---

## 성공 기준

- 사용자가 예전 작업 세션을 10초 안에 다시 찾을 수 있다
- 특정 키워드로 원하는 메시지 문맥을 바로 열 수 있다
- Memradar가 단순 시각화가 아니라 "기록 도구"처럼 느껴진다

---

## 구현 우선순위

### Phase A

- 검색 입력창
- 풀텍스트 검색
- 메시지 단위 결과
- 세션 이동
- 검색어 하이라이트

### Phase B

- 모델/툴/프로젝트/날짜 필터
- 정렬 옵션
- URL 쿼리 연동

### Phase C

- IndexedDB 기반 검색 캐시
- 결과 그룹핑 고도화
- 성능 최적화
