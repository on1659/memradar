# Memradar UI/UX Principles

> 현재 구현(`src/index.css`, `src/components/*`, `src/theme/*`, `src/i18n.tsx`)을 기준으로 역으로 추출한 운영 원칙 문서.
> 새 방향을 제안하는 문서가 아니라, "지금 UI가 실제로 따르는 규칙"을 정리한 문서다.

## 한 줄 정의

Memradar의 기본 UI는 차분한 분석 대시보드이고, 보조 경험은 감성적인 스토리텔링 레이어다. 평소에는 도구처럼 빠르고, 회고할 때는 작품처럼 몰입하게 만든다.

## 기준 용어

- `첫 화면`: 파일이 아직 로드되지 않았을 때 보이는 시작 화면
- `대시보드`: 세션이 로드된 뒤 기본으로 보는 메인 분석 화면
- `검색`: 별도 검색 작업면
- `세션 상세`: 개별 대화 흐름을 보는 화면
- `성향 분석`: 코딩 성향과 사용 유형을 해석하는 화면
- `Code Report`: 회고와 공유를 위한 풀스크린 스토리 화면

이 문서 본문에서는 위 명칭만 사용한다.

## 제품 태도

- CLI가 1순위다. 첫 화면도 드래그 앤 드롭보다 `npx memradar` 복사를 먼저 권한다.
- UI는 원본 로그를 덮는 장식이 아니라, 로그를 더 빨리 이해하게 만드는 해석층이어야 한다.
- 대시보드는 "분석 툴"처럼 침착해야 하고, Code Report는 "회고 경험"처럼 감정선을 강화해도 된다.
- 정보가 많아도 복잡해 보이면 안 된다. 그래서 카드, 필터, 오버레이, 보조 정보가 모두 단계적으로 열린다.
- 사용자는 오래 배울 필요 없이 바로 행동할 수 있어야 한다. 복사, 열기, 검색, 공유, 테마 변경은 한두 번의 액션 안에 끝난다.

## 핵심 디자인 철칙

### 1. 주 행동을 먼저 보여준다

- 첫 화면의 추천 경로는 "폴더 선택"이 아니라 "명령어 복사 후 실행"이다.
- 보조 경로는 존재하지만, 추천 경로보다 앞에 두지 않는다.
- 사용자가 무엇을 해야 할지 읽지 않아도 알 수 있어야 한다.

### 2. 대시보드는 차분하고, 강조는 액센트로만 준다

- 기본 배경은 저채도 다크/라이트 계열이고, 시선 유도는 액센트 컬러와 미세한 광택으로 처리한다.
- 화려함은 전체가 아니라 포인트에만 쓴다.
- CTA, 주요 수치, 선택 상태, 최신성 배지는 액센트가 담당한다.

### 3. 카드는 "덩어리"가 아니라 "단위"로 보이게 한다

- 카드끼리 시각적으로 붙어 보이게 하지 않는다.
- 같은 행에 있더라도 각 카드는 독립된 경계, 반경, 패딩을 유지한다.
- 정보 구조는 합쳐 보이게 만드는 대신, 가까운 간격과 반복 규칙으로 묶는다.

### 4. 요약 카드와 시각화 카드는 정렬 방식이 다르다

- 요약 카드, 컴팩트 카드, 보조 카드의 첫 정보는 제목 바로 아래에 붙어 있어야 한다.
- 히트맵, 시간대 차트, 도넛 차트처럼 "보는 카드"는 본문 영역 중앙에 놓는다.
- 작은 카드 안에서 내용을 억지로 수직 중앙 정렬하지 않는다.

### 5. 한 화면에서 모든 제어를 한꺼번에 드러내지 않는다

- 검색은 쿼리 입력이 먼저고, 필터는 접힌 상태에서 필요할 때만 연다.
- 테마 변경은 한 번에 전부 고르는 대신, 모드와 액센트를 2단계로 분리한다.
- 업데이트, 테마 패널, 공유 메뉴는 모두 오버레이/팝오버로 분리해 메인 레이아웃을 흐리지 않는다.

### 6. 호버는 "추가 정보"를 주지만, 본 기능을 가두지 않는다

- 툴팁, 비용 패널, 차트 상세값, 검색 하이라이트는 이해를 돕는 보조 정보다.
- 핵심 경로는 클릭, 키보드, 명시적 버튼으로도 반드시 닿아야 한다.
- 호버는 정답이 아니라 보강재다.

### 7. 모션은 작고 부드럽고 반복 주기가 길어야 한다

- 전체 레이아웃을 흔드는 큰 모션 대신, 경계광, 아이콘 미세 이동, 카드 등장 같은 작은 애니메이션을 쓴다.
- 주기성 애니메이션은 CTA를 환기시키되 화면이 늘 분주해 보이게 만들면 안 된다.
- `prefers-reduced-motion`에서는 비필수 애니메이션을 꺼야 한다.

### 8. 테마는 컴포넌트가 아니라 시스템에서 관리한다

- 배경, 카드, 보더, 텍스트, 액센트는 CSS 변수와 프리셋으로 통제한다.
- 새 색은 임의의 컴포넌트 안에 흩뿌리지 않고 `src/theme/themePresets.ts`와 `src/index.css` 기준으로 넣는다.
- 테마가 바뀌어도 구조와 정보 계층은 유지되어야 한다.

### 9. 대시보드와 Code Report는 같은 제품의 다른 무드다

- 대시보드는 정보 탐색, Code Report는 감정적 회고와 공유를 담당한다.
- Code Report는 독립 팔레트와 전용 타이포, 풀스크린 내러티브를 사용한다.
- 대신 탐색 규칙은 단순해야 한다. 클릭으로 다음 장, 명확한 진행바, 이전/다음, 마지막 슬라이드의 탈출 경로가 있어야 한다.

### 10. 정보는 "원자료"보다 "해석"이 먼저 와야 한다

- 사용 모델, 활동 시간, 성향, 사용 카테고리는 모두 원본 로그를 재가공한 요약이다.
- 결과는 전체를 나열하기보다 상위 항목, 대표 수치, 짧은 라벨 중심으로 보여준다.
- 원자료가 길면 세션 상세에서만 길게 풀고, 메인 뷰에서는 압축해서 보여준다.

### 11. 텍스트는 한국어 중심이되, 영어 사용자도 바로 이해할 수 있어야 한다

- 로케일은 `ko`/`en` 이중 구조로 관리한다.
- 한국어 화면에서도 모델명, 기술 용어, CTA 일부는 영어를 자연스럽게 섞는다.
- 브랜드 톤은 지나치게 딱딱하지 않고, 약간의 에디토리얼 감성과 제품 친화성을 유지한다.

### 12. 모바일에서도 읽는 순서가 깨지면 안 된다

- 데스크톱에서는 다단 그리드를 쓰되, 모바일에서는 한 열로 자연스럽게 접힌다.
- 줄어든 화면에서 가장 먼저 보여야 할 정보는 변하면 안 된다.
- 오버레이와 팝오버도 모바일에서 다시 위치를 잡아야 한다.

## 화면별 패턴

### 첫 화면

- 추천 경로를 가장 크게 보여준다.
- "복사 가능한 명령어" 자체를 UI의 핵심 오브젝트로 다룬다.
- 신뢰 요소는 별도 카드로 빼서 안심감을 만든다.
- 폴더 선택은 보조 경로지만, 드래그 앤 드롭과 직접 선택 둘 다 허용한다.

### 대시보드

- 정보 흐름은 요약 수치 → 활동 패턴 → 분석 인사이트 → 세션 리스트 순서다.
- 가장 위는 브랜드와 주요 진입점만 둔다. 보조 기능은 아이콘 버튼과 팝오버로 눌러 담는다.
- 토큰/비용처럼 복합 데이터는 하나의 카드 안에서 스위치와 툴팁으로 점진적으로 푼다.
- 세션 목록은 "필터와 정렬"을 먼저 두고, 검색은 그 아래에서 작동하게 한다.

### 검색

- 검색은 별도 작업면이다. 대시보드 안의 작은 위젯처럼 다루지 않는다.
- 빈 상태에서는 사용법을 보여주고, 결과 상태에서는 스니펫 중심으로 빠르게 훑게 한다.
- 필터는 감춰두고, 결과 개수와 정렬은 상단에 얇게 붙인다.
- 결과 아이템은 "문맥 스니펫 → 메타 정보 → 원 세션 힌트" 순으로 보여준다.

### 세션 상세

- 긴 대화는 바로 본문으로 내려가지 않고, 먼저 세션 메타 요약 카드로 맥락을 잡는다.
- 메시지 스트림은 역할 색을 분리하되, 전체 화면의 톤은 유지한다.
- 강조된 메시지가 있으면 스크롤로 직접 데려가고, 시각적 하이라이트를 준다.
- 너무 긴 텍스트는 잘라서 보여주고, 목록 탐색성이 깨지지 않게 한다.

### 성향 분석

- 분석 결과를 설명형 카드로 번역하는 화면이다.
- 진지한 데이터 위에 살짝 놀이적인 해석을 얹지만, 구조는 여전히 카드 기반이다.
- 대표 유형은 크게, 나머지 유형은 그리드로 비교하게 만든다.

### Code Report

- 여기서는 "대시보드 규칙"보다 "한 장면 한 메시지"가 더 중요하다.
- 풀스크린, 강한 여백, 큰 타이포, 전용 다크 스토리 팔레트를 사용한다.
- 이동 규칙은 단순해야 한다. 클릭/방향키/진행 점/스킵 버튼이 모두 같은 흐름을 가리켜야 한다.
- 마지막은 공유 가능 이미지로 닫는다. 즉, 감정선의 끝이 곧 행동 유도여야 한다.

## 비주얼 시스템 요약

### 컬러

- 배경 테마는 `dark`, `night`, `light`, `paper` 네 축으로 운영한다.
- 액센트는 `indigo`, `teal`, `rose`, `amber`, `violet` 다섯 축으로 운영한다.
- 의미 색은 반복적으로 고정 사용한다.
- `green`: 성공, 사용자 쪽, 성장, 긍정 지표.
- `amber`: 토큰, 하이라이트, 주목 정보.
- `rose`: 경고, 바쁜 날, 강한 포인트.
- `cyan`: 보조 분석, 패턴, 서브 인사이트.

### 타이포그래피

- 본문은 가독성 좋은 산세리프 기반이다.
- 헤드라인은 세리프 계열을 섞어 "리포트"와 "에디토리얼" 감각을 만든다.
- 숫자는 수치 카드에서 매우 중요하므로 크게, 짧게, 빠르게 읽히게 둔다.

### 표면과 반경

- 카드 반경은 둥글지만 과장되지 않는다.
- 표면은 단색보다 미세한 투명도와 대비 차이로 레이어를 만든다.
- 진한 그림자보다 얕은 보더와 약한 내부 광택을 더 자주 쓴다.

### 데이터 시각화

- 라벨은 가급적 가로로 읽히게 둔다.
- 축, 툴팁, hover 상태는 공통 톤을 공유해야 한다.
- 차트는 "예쁘게"보다 "즉시 읽히게"가 우선이다.
- 상위 몇 개 항목만 드러내고, 자세한 값은 hover나 캡션으로 보강한다.

## 구현 가드레일

- 대형 패널은 로컬 스태킹에 기대지 말고 포털/최상위 레이어로 띄운다.
- 새 테마 색을 넣을 때는 컴포넌트 안에 직접 하드코딩하지 않는다.
- 컴팩트 카드의 첫 수치가 제목 아래에서 멀어지지 않게 한다.
- 히트맵/차트 라벨을 회전시켜서 공간을 억지로 해결하지 않는다.
- 대시보드 팔레트와 Code Report 전용 팔레트를 섞지 않는다.
- 강조 애니메이션은 모든 버튼에 쓰지 않는다. 정말 중요한 CTA에만 쓴다.
- hover 없이도 핵심 행동이 가능해야 한다.
- 필터나 설정은 처음부터 전부 펼치지 않는다.

## 새 UI 작업 체크리스트

- 이 화면의 1순위 행동이 첫 시선 안에 들어오는가
- 카드가 서로 붙어 보이지 않고 독립 단위로 읽히는가
- 컴팩트 카드와 시각화 카드의 정렬 규칙이 섞이지 않았는가
- 액센트 컬러가 강조에만 쓰이고 있나
- hover 없이도 주요 기능이 가능한가
- 오버레이가 다른 카드 뒤로 깔리지 않는가
- 모바일에서 읽는 순서와 우선순위가 유지되는가
- `prefers-reduced-motion`에서 경험이 깨지지 않는가
- 대시보드와 Code Report의 무드가 서로 침범하지 않는가
- 새 화면이 "분석 도구"인지 "스토리텔링 화면"인지 성격이 명확한가
