# Memradar Deployment Guide

> Memradar을 다른 사람이 실제로 사용할 수 있게 배포하는 방법 정리

## TL;DR

현재 Memradar은 **서버 없이 동작하는 정적 웹앱**이다. 가장 적합한 기본 배포 방식은 **Vercel 또는 GitHub Pages에 정적 배포**하고, 사용자가 브라우저에서 자신의 `.jsonl` 파일을 직접 불러오게 하는 방식이다.

이 방식이 현재 아키텍처와 가장 잘 맞는다.

- 장점: 백엔드 불필요, 비용 낮음, 배포 간단, 프라이버시 친화적
- 제약: 브라우저가 사용자 PC의 `~/.claude/projects/` 폴더를 자동으로 읽을 수는 없음
- 결론: 지금은 `배포된 URL + 사용자의 수동 파일 선택/드래그`가 기본 UX

## 현재 구조 기준 사실 정리

- 앱은 `Vite`로 빌드되는 정적 프론트엔드다.
- 현재 배포 설정은 Vercel 정적 배포를 기준으로 잡혀 있다.
- 세션 파싱은 브라우저 안에서 이뤄진다.
- 현재 파일 입력 UX는 드래그 앤 드롭 또는 파일 선택이다.
- 현재 등록된 프로바이더는 `Claude Code` 하나다.

## 배포 방식 옵션

### 1. 정적 웹 배포

가장 추천하는 현재 기본안.

### 개요

- `npm run build`로 생성된 `dist/`를 정적 호스팅에 배포
- 사용자는 URL에 접속한 뒤 `.jsonl` 파일 또는 폴더를 직접 불러옴
- 분석은 브라우저에서 수행

### 적합한 서비스

- Vercel
- GitHub Pages
- Netlify
- Cloudflare Pages

### 장점

- 현재 코드와 가장 잘 맞음
- 서버 비용이 거의 없음
- 배포가 매우 단순함
- 사용자의 로그가 서버에 업로드되지 않음

### 단점

- 처음 사용할 때 사용자가 파일을 직접 선택해야 함
- 로컬 폴더 자동 스캔은 불가
- 브라우저별 파일 시스템 API 제약이 있음

### 추천 상황

- 오픈소스 프로젝트 공개
- 데모 사이트 운영
- 개인 사용자가 직접 자신의 로그를 분석

### 권장도

**매우 높음. 현재 Memradar의 공식 배포 방식으로 삼기 가장 좋다.**

### 2. 정적 웹 + 폴더 연결 UX

정적 웹 배포를 유지하면서 드래그 의존성을 줄이는 확장안.

### 개요

- 브라우저의 File System Access API를 활용
- 사용자가 처음 한 번 `Claude 로그 폴더`를 선택
- 이후에는 앱이 저장된 핸들을 통해 다시 접근 시도

### 장점

- UX가 훨씬 좋아짐
- 드래그 없이 `폴더 연결` 버튼으로 사용 가능
- 현재 정적 웹 구조를 유지할 수 있음

### 단점

- 첫 1회 권한 허용은 여전히 필요
- 브라우저 지원 범위가 제한적일 수 있음
- 완전 무권한 자동 접근은 여전히 불가

### 추천 상황

- 웹앱 구조를 유지하고 싶음
- 사용자 경험을 지금보다 더 앱처럼 만들고 싶음

### 권장도

**높음. 다음 단계 개선안으로 가장 현실적이다.**

### 3. GitHub에 로그를 올리고 URL로 불러오기

데모 용도로는 가능하지만, 실제 개인 사용 로그용 기본안으로는 권장하지 않음.

### 개요

- `.jsonl` 파일을 GitHub 저장소에 업로드
- 앱이 Raw GitHub URL에서 파일을 fetch해서 분석

### 장점

- 파일 드래그 없이 URL만으로 불러오기 가능
- 샘플 데이터 공유가 쉬움
- 데모 링크 제작에 적합

### 단점

- 공개 저장소에 올리면 세션 로그가 외부에 노출될 수 있음
- 비공개 저장소는 브라우저 단독으로 안전하게 접근하기 어려움
- GitHub 토큰이 필요한 경우 프론트엔드 보안 문제가 생김

### 추천 상황

- 샘플 세션 공개 데모
- 팀 내부에서 비민감 테스트 데이터 공유

### 권장도

**낮음. 개인 실제 로그에는 보안상 기본 배포 방식으로 추천하지 않는다.**

### 4. 백엔드 업로드형 웹 서비스

로그를 서버에 업로드하고 계정 단위로 저장/조회하는 방식.

### 개요

- 사용자가 `.jsonl` 파일을 서버에 업로드
- 서버 또는 스토리지에 보관
- 이후에는 자동으로 다시 조회 가능

### 장점

- 진짜 웹서비스처럼 동작 가능
- 여러 기기에서 같은 데이터 접근 가능
- 자동 동기화/공유 기능 추가가 쉬움

### 단점

- 현재 제품 철학과 어긋남
- 개인정보 및 보안 부담이 크게 증가
- 백엔드, 인증, 저장소, 삭제 정책이 모두 필요

### 추천 상황

- SaaS 제품으로 확장할 계획이 분명함
- 팀 단위 협업/공유가 핵심 요구사항임

### 권장도

**낮음. 현재 Memradar의 방향과는 거리가 있다.**

### 5. 데스크톱 앱 배포

브라우저 제약 없이 로컬 파일 접근 UX를 크게 개선하고 싶을 때 고려할 수 있는 방식.

### 개요

- Tauri 또는 Electron으로 패키징
- 사용자가 앱을 설치해서 사용
- 로컬 파일 접근을 웹보다 자연스럽게 처리 가능

### 장점

- 로컬 폴더 자동 탐색 UX에 유리
- 브라우저 보안 제약이 적음
- 오프라인 앱 경험 제공 가능

### 단점

- 설치형 제품이 됨
- 배포/업데이트/서명 등 관리 비용 증가
- 현재 로드맵의 "웹 전용" 방향과 다를 수 있음

### 추천 상황

- 진짜 자동 불러오기 경험이 중요함
- 로컬 우선 앱 경험이 제품 핵심임

### 권장도

**중간. 필요가 명확할 때만 검토. 현재 기본안은 아님.**

## 추천 결론

현재 Memradar에는 아래 전략이 가장 적합하다.

### 지금 당장

- **1차 배포는 Vercel 기반 정적 웹 배포**
- 사용자는 URL 접속 후 `.jsonl` 파일을 직접 불러온다

### 다음 개선

- **`폴더 연결` 버튼을 추가**
- 드래그 없이 첫 1회 권한 허용 후 재사용 가능하게 만든다

### 하지 않는 것을 권장

- 개인 실제 로그를 공개 GitHub에 올려서 쓰는 방식
- 초기 단계에서 백엔드 업로드형 서비스로 확장하는 것

## 권장 배포 시나리오

### 시나리오 A. 공개용 오픈소스/데모

가장 권장되는 운영 방식.

1. GitHub 저장소에 코드 푸시
2. Vercel에 연결
3. 자동 배포
4. 사용자에게 URL 제공
5. 사용자는 자신의 로그 파일을 직접 드래그하거나 선택

### 사용자 안내 문구 예시

```text
Memradar은 브라우저에서 Claude Code 세션 로그를 분석합니다.
로그는 서버로 업로드되지 않으며, 사용자의 PC에서 직접 불러와 브라우저 안에서 처리됩니다.
Claude Code 로그 폴더에서 .jsonl 파일을 선택하거나 드래그해 시작하세요.
```

### 시나리오 B. 샘플 데이터 데모

체험용 랜딩 페이지나 데모에 적합.

1. 공개 가능한 샘플 `.jsonl` 준비
2. GitHub 공개 저장소 또는 정적 파일 경로에 업로드
3. 앱에 `샘플 불러오기` 버튼 추가
4. 버튼 클릭 시 공개 URL에서 fetch

### 주의

- 실제 개인 로그에는 사용하지 않는 것이 좋다
- 민감한 프롬프트/코드/경로가 들어갈 수 있다

### 시나리오 C. 드래그 없는 개선형 웹앱

정적 배포를 유지하면서 UX를 높이는 중기안.

1. Vercel 정적 배포 유지
2. `폴더 연결` 버튼 추가
3. 폴더 핸들을 브라우저 저장소에 보관
4. 재접속 시 권한 상태 확인 후 다시 로드

### 기대 효과

- 사용자는 거의 설치형 앱처럼 느낄 수 있음
- 다만 완전 자동 접근이 아니라 브라우저 권한 모델 안에서 동작함

## Vercel 배포 메모

현재 저장소에는 이미 Vercel 설정 파일이 있다.

```json
{
  "buildCommand": "npm run build",
  "outputDirectory": "dist",
  "framework": "vite"
}
```

따라서 배포 실무는 아래 흐름이면 충분하다.

1. GitHub에 저장소 푸시
2. Vercel에서 저장소 import
3. Framework preset 확인
4. Build command가 `npm run build`인지 확인
5. Output directory가 `dist`인지 확인
6. 배포 완료 후 URL 공유

## 사용자 운영 가이드에 포함하면 좋은 문구

- 현재 지원 로그 형식은 `Claude Code .jsonl`
- 기본 경로 예시:
  - macOS/Linux: `~/.claude/projects/`
  - Windows: `%USERPROFILE%\\.claude\\projects\\`
- Memradar은 기본적으로 로컬 로그를 브라우저에서 직접 읽는다
- 브라우저 보안 정책상 사용자의 로컬 폴더를 무권한으로 자동 스캔할 수는 없다

## 최종 제안

Memradar의 공식 배포 전략은 아래처럼 정의하는 것이 가장 자연스럽다.

1. **공식 배포 방식은 정적 웹 배포**
2. **기본 데이터 입력 방식은 로컬 `.jsonl` 파일 불러오기**
3. **차기 UX 개선으로 `폴더 연결` 기능 검토**
4. **GitHub 로그 import는 샘플 데이터 데모 전용으로만 사용**
5. **백엔드 업로드형과 데스크톱 앱은 제품 방향이 바뀔 때 재검토**
