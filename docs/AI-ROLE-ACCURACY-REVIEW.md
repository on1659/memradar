## 왜 "정확하다"고 보기 어려운가

### 1. `includes` 기반 substring 매칭이라 오탐 여지가 크다

핵심 문제는 `text.includes(keyword)` 를 그대로 쓴다는 점이다.

이 방식은 단어 경계를 보지 않기 때문에, 짧은 키워드가 다른 단어 내부에 들어가도 매치된다.

대표 예시:

- `design` 카테고리 키워드에 `ui` 가 있다.
- `feature`/`devops` 문맥에서 자주 나오는 `build` 는 문자열 내부에 `ui` 를 포함한다.
- 따라서 `"build"` 가 들어간 문장이 `design` 으로도 잡힐 수 있다.

비슷한 종류의 오탐 가능성:

- `add` → `address`
- `text` → 거의 모든 문서/설명 문맥
- `read` → 문맥에 따라 일반 동사로 넓게 사용

즉, 현재 점수는 "정확한 의미 분류"가 아니라 **짧은 문자열이 우연히 겹친 결과**를 일부 포함할 수 있다.

### 2. 키워드 자체가 너무 넓은 항목이 섞여 있다

`src/lib/usageProfile.ts:23-79` 기준으로, 아래 키워드는 역할을 좁게 특정하기보다 일반 문장 습관을 반영할 가능성이 있다.

- `review`: `how`, `what`, `read`
- `writing`: `text`, `email`
- `devops`: `build`, `server`, `config`, `github`, `action`
- `data`: `model`, `fetch`
- `feature`: `add`

예를 들어:

- `"how do we fix this?"` 는 review + debug 둘 다 쉽게 잡힐 수 있다.
- `"build fails on Vercel"` 은 feature + debug + devops + design(`ui`)까지 겹칠 여지가 있다.
- `"read this file"` 같은 표현은 의도상 단순 탐색인데 review 점수를 밀어 올릴 수 있다.

### 3. 한 메시지가 여러 직업에 동시에 점수될 수 있다

현재 로직은 한 메시지가 여러 카테고리에 중복 기여하는 구조다.

예:

- "에러 원인 분석하고 API 연결 코드 수정해줘"

위 요청 하나만으로도 다음이 동시에 올라갈 수 있다.

- `debug`
- `review`
- `feature`
- 경우에 따라 `data`

따라서 차트의 막대 길이는 "서로 배타적인 역할 비중"이 아니라, **겹칠 수 있는 키워드 hit 누적값**에 더 가깝다.

### 4. 최소 표본 / 판별 유보 / 신뢰도 개념이 없다

현재는 카테고리가 한 번이라도 잡히면 순위가 생긴다.

없거나 약한 것들:

- 최소 메시지 수 기준
- 표본이 적을 때의 "판별 유보"
- 상위 1, 2위가 거의 비슷할 때의 명시적 신뢰도
- "이 결과는 참고용 추정치"라는 구조적 장치

그래서 실제로는 데이터가 얇아도 사용자 입장에서는 "대표 직업이 판정됐다"처럼 느껴질 수 있다.

## 정확도에 대한 제품 관점 판단

현재 구현은 아래 문장에는 비교적 맞다.

- "사용자 요청에서 자주 나온 표현을 바탕으로, AI가 많이 맡은 역할을 가볍게 요약한다."
- "플레이풀한 리캡 요소로는 쓸 수 있다."

하지만 아래 문장에는 아직 무리가 있다.

- "이 사용자의 AI 활용 직업을 정확하게 분류한다."
- "대표 역할을 신뢰도 높게 판정한다."
- "서로 유사한 역할까지 안정적으로 구분한다."

즉, **fun insight 로는 가능하지만, 정확한 taxonomy 로 포장하기엔 아직 거칠다** 는 쪽이 현재 코드와 더 맞는 해석이다.
